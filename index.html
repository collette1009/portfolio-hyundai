Case 01

배송 제약 VOC를 Fail-soft UX로 전환 — 개발 구현 제약 속 고객 혼선 최소화

Problem

배송 가능 요일이 다른 상품이 장바구니에 함께 담기면 주문이 차단됐지만, 고객에게는 기존 품절형 메시지만 노출되어 실제 원인을 알기 어려웠습니다. 고객은 이를 품절이나 시스템 오류로 인식했고, 동일한 문의가 일 6~10건 반복됐습니다.

핵심 문제는 고객의 이해 부족이 아니라, 주문이 막히는 시점에 시스템이 실제 차단 사유를 설명하지 못하는 UX 구조에 있었습니다.

A Fail-soft UX 설계

실제 원인은 상품별 배송 가능 요일이 다른 데 있었지만, 배송 가능 요일·상품 조합·도착 가능일을 모두 반영해 차단 로직을 전면 개편하기에는 개발 구현 난도가 높고 영향 범위가 큰 상황이었습니다.

그래서 차단 로직 자체를 바꾸기보다, 장바구니 단계에서 상품별 배송 가능 요일을 subtext로 함께 안내하는 방식을 제안했습니다. 고객이 차단 메시지를 보더라도 상품 옆 안내로 원인을 확인하고, 필요한 경우 상품을 분리하거나 제외해 주문을 조정할 수 있도록 한 것입니다.

또한 특정 상품에 한정되지 않고 다른 주의상품에도 재사용할 수 있도록, 라벨을 상품별로 다르게 적용 가능한 구조를 함께 제안했습니다. 데이터 자료와 함께 개발팀에 공유했고, 제안은 즉시 수용되어 당일 구현됐습니다. 이후 유사한 배송 제약이 있는 상품에도 같은 구조로 확장 적용됐습니다.

Result

반복 문의는 일 6~10건 수준에서 0건 수준으로 감소했습니다. 고객이 품절이나 시스템 오류로 오인하던 문제를, 상품별 배송 가능 요일 안내를 통해 원인을 확인할 수 있는 구조로 바꾼 결과입니다.

일 6~10건 → 0건 수준
반복 문의 해소 · 개발팀 제안·당일 구현 · 유사 케이스로 확장 적용

실제 화면

Insight

· 같은 정보를 충분히 안내하지 않을 때, 고객은 시스템 오류로 인식합니다.

· 차단 로직 전체를 바꾸기 어려운 상황에서도, 막히는 시점에 이유를 보여주는 fail-soft UX로 고객 혼선을 줄일 수 있습니다.

· 고객 접점에서 발견한 반복 패턴은 개발 구현 범위와 운영 현실을 함께 고려할 때 실제 개선으로 연결될 수 있습니다.

Case 02

고객 의도 기반 환불·교환 입력 구조 재설계 — 재확인 왕복 해소

Problem

환불·교환 등 고객이 원하는 해결방법이 별도 선택값으로 분리되어 있지 않아, 자유 텍스트 안에서 고객 의도를 다시 확인해야 하는 구조였습니다.

고객 의사 확인 없이 임의 처리할 경우 분쟁 리스크가 있어, 문의 접수 단계에서 고객이 원하는 해결방법을 먼저 선택하도록 하는 구조가 필요했습니다.

A 1:1 문의 접수 단계 '원하는 해결방법' 선택 필드 신설

1:1 문의 화면에 '원하는 해결방법' 선택 필드를 신설해 고객이 문의 작성 시 환불/교환 의사를 먼저 선택하도록 하고, 해당 값이 CRM의 해결방법 필드로 노출되도록 개선안을 정리했습니다.

B 기존 CRM 영역 활용 · 개발 범위 최소화

답변 작성일과 중복되는 '확인일시' 영역을 '해결방법' 필드로 대체하는 방향을 제안해, 별도 개발 범위를 늘리지 않고 구조 변경이 가능한 지점을 확인했습니다.

Result

고객 의도 확인을 위한 추가 답변 왕복을 줄이고, 담당자가 문의 접수 단계에서 해결방법 선택값을 바로 확인할 수 있어 후속 처리 기준이 더 명확해졌습니다.

재확인 왕복 해소 · 처리 기준 명확화
자유 텍스트 → 해결방법 선택값 · CRM 필드 연결 · 개발 범위 최소화

실제 화면

Insight

· 응대 단계에서 발생하는 재확인 비용은 대부분 입력 단계의 구조 문제에서 시작됩니다.

· 자유 텍스트는 유연해 보이지만, 실제로는 의도를 흐리는 구조입니다.

· 큰 개발 변경 없이 기존 필드 활용 방식만 다시 설계해도 운영 비용을 빠르게 줄일 수 있습니다.

Case 03

카카오VX 400+ 제휴사 정책 정보 구조 표준화 — 운영 기준을 동일하게 찾고 판단할 수 있도록

Problem

골프장 400여 곳의 예약·취소·위약 기준이 제휴사별로 다르게 관리되고 있었습니다. 어드민에 데이터는 있었지만, 골프장마다 정책의 표현 방식과 정보 배열 순서가 달라 담당자가 필요한 정보를 빠르게 찾기 어려운 구조였습니다.

예를 들어 어떤 골프장은 위약 기간 내 취소 시 예약만 제한되는지, 동반자로도 방문이 제한되는지까지 명확히 적혀 있었지만, 어떤 골프장은 표현이 모호했습니다. 위약금 입금 계좌, 예약실 전화번호, 운영시간, 취소 가능 기한 같은 핵심 정보도 골프장마다 위치가 달라, 동일한 문의를 처리할 때마다 전체 정책 문구를 다시 훑어야 했습니다.

그 결과 담당자별로 확인 속도와 해석 방식에 차이가 생겼고, 신규 인력이 정책을 익히는 데도 시간이 걸렸습니다.

A 400여 개 제휴사 정책 정보 구조 표준화

400여 개 제휴 골프장의 정책을 전수 확인하고, 담당자가 실제 문의 처리 시 먼저 확인해야 하는 순서대로 정보 구조를 재정리했습니다.

예약실 전화번호와 운영시간, 취소 가능 기한, 위약 적용 범위, 위약금 및 입금 계좌, 동반자 제한 여부, 예외 기준처럼 자주 확인하는 항목을 공통 구조로 정렬했습니다. 제휴사마다 다르게 쓰이던 표현도 동일한 기준으로 바꿔, 담당자가 같은 위치에서 같은 유형의 정보를 확인할 수 있도록 정리했습니다.

B 취소·위약·동반자 변경 기준 재정리

단순히 정보를 옮겨 적는 것이 아니라, 반복 문의가 발생하는 기준을 중심으로 정책을 다시 분류했습니다. 특히 위약 기간 내 취소 시 이후 예약만 제한되는지, 동반자 방문까지 제한되는지처럼 고객 안내에 영향을 주는 항목을 별도 기준으로 정리했습니다.

이를 통해 담당자가 전체 위약 규정을 매번 처음부터 읽지 않아도, 필요한 판단 항목을 빠르게 확인할 수 있도록 운영 가이드와 Knowledge Base를 재정비했습니다.

C 정책 최신화 및 온보딩 기준 정비

정책 변경이 발생할 때는 어드민과 내부 문서의 반영 위치가 달라지지 않도록 공유 흐름을 정리했습니다. 신규 인력에게도 동일한 순서와 표현 기준으로 안내할 수 있도록 온보딩 자료에 반영해, 개인 숙련도에 따라 판단이 달라지는 상황을 줄이고자 했습니다.

Result

제휴사별로 흩어져 있던 정책 정보가 공통 순서와 표현 기준으로 정리되면서, 담당자가 필요한 정보를 더 빠르게 찾고 동일한 방식으로 판단할 수 있는 기반이 마련되었습니다.

400+ 제휴사 정책 정보 구조 표준화
정보 순서·표현 기준 통일 · 정보 탐색 시간 단축 · 온보딩 기준 정비

Insight

· 흩어진 정보는 자동으로 기준이 되지 않습니다. 실제 처리 순서에 맞게 재배열되어야 운영 기준이 됩니다.

· 정책 표준화는 내용을 하나로 통일하는 일이 아니라, 서로 다른 정책을 같은 방식으로 찾고 판단할 수 있게 만드는 일입니다.

· 파트너 운영에서 중요한 것은 모든 제휴사를 같게 만드는 것이 아니라, 다른 정책도 동일한 구조로 이해할 수 있게 만드는 것입니다.

Case 04

외부 물류 파트너 사고·배상 기준 정리 — 택배사 증빙 기준 간소화와 물류 운영사 전환 초기 비용 반영 구조 안정화

Problem

배송 사고가 발생했을 때 택배사와 외부 물류 운영사, 즉 3PL 파트너의 사고 확인 흐름, 증빙 기준, 비용 반영 기준이 각각 달라 배상 회수와 후속 처리가 지연되는 문제가 있었습니다.

택배사 사고 처리에서는 회수 필수, 송장 사진·외부 박스 사진·내품 사진 등 3종 증빙 요구로 인해 고객 응대 현장에서 자료 확보가 어려웠습니다. 주말 이후 접수된 건은 이미 상품이 폐기된 경우가 있었고, 유리병·간장류처럼 파손 시 안전 문제가 있는 상품은 고객에게 내품 사진을 다시 요청하기 어려웠습니다. 이로 인해 실제 사고가 확인된 건도 배상 처리로 이어지기 어려웠고, 배상 회수액은 월 50만 원 수준에 머물러 있었습니다.

이후 물류 운영사 전환 초기에는 오피킹, 출고누락, 포장불량, 파손 등 파트너 귀책 가능성이 있는 사고가 반복됐습니다. 그러나 전환 초기라 사고 확인, 책임 판단, 비용 반영 기준이 안정적으로 자리 잡지 못한 상태였고, 파트너 회신이 늦어지면 고객 환불·재배송·CS 보상 비용이 내부 부담으로 남는 구조였습니다.

A 택배 표준약관 근거 제시 · 회수·사진 증빙 기준 간소화

택배사 파손 사고 처리의 병목은 회수와 고객 증빙 기준이었습니다. 저는 단순히 회수 기준 완화를 요청하기보다, 택배 표준약관상 운송 중 포장 훼손이나 사고 인지 시 송화인에게 통보하고 조치해야 한다는 점에 주목했습니다.

파손 스캔이나 운송 중 훼손이 택배사 시스템상 확인된 건은 배송 보류 후 송화인에게 먼저 공유하고 처리 방향을 확인해야 하며, 이미 배송이 진행되었거나 회수가 어려운 경우에는 고객에게 회수나 내품 사진을 다시 요구하기보다 택배사 사고 접수 기록과 현장에서 확보 가능한 외부 박스 파손·오염 사진을 기준으로 처리할 수 있도록 논점을 정리했습니다.

이후 로젠 측에서도 표준약관상 사고 발생 시 조치 의무를 인정했고, 내부 검토 후 사고 처리 기준을 재정리했습니다. 그 결과 파손 스캔 등 택배사 시스템상 사고가 확인된 건은 회수·사진 3종 없이 사고접수가 가능해졌고, 스캔이 없는 건도 필요 시 회수하되 외부 박스 파손·오염 사진 등 현장 확인 가능한 증빙을 기준으로 사고접수를 진행할 수 있게 되었습니다.

이를 통해 고객에게 반복적으로 회수와 사진 3종을 요청해야 하는 부담을 줄였고, 배상 회수액은 월 50만 원 수준에서 약 700만 원 수준으로 개선되었습니다.

B 물류 운영사 전환 초기 계약서 기반 물류비 차감 기준 정리

물류 운영사 전환 이후에는 사고 발생 건을 단순 요청으로만 전달하면 파트너 회신 지연으로 후속 처리가 멈추고, 책임 귀속이 흐려질 수 있었습니다.

이에 물류 계약서 원문을 먼저 확인해, 상차 이후 발생한 파손·누락·운송 중 사고와 관련 비용 부담이 물류 파트너 책임으로 정리될 수 있는 조항을 확인했습니다. 이를 바탕으로 초록마을이 먼저 처리한 환불액, 재배송비, CS 보상 비용을 익월 물류비 정산에서 차감 반영할 수 있는 구조로 정리했습니다.

피해 유형, 비용 항목, 정산 확인 기준을 정리해 유관 리더에게 공유했고, 피해 현황 통보서와 정산 참고용 집계 파일을 작성했습니다. 이후 온도대, 주문 단위, 출고 전·출고 후 사고 구분 등 산정 기준을 보완해 실제 물류비 차감 기준으로 운영에 반영했습니다.

계약서 기준과 정산 반영 가능성을 파트너사에도 명확히 전달한 이후, 사고 확인과 회신 속도가 이전보다 개선되었고 후속 처리 흐름도 안정화되었습니다.

Result

택배사 사고 처리에서는 과도한 회수·고객 증빙 기준으로 막혀 있던 배상 회수 흐름을 실무 증빙 기준으로 재정리해, 배상 회수액을 월 50만 원 수준에서 약 700만 원 수준으로 개선했습니다.

물류 운영사 전환 이후에는 계약서 원문을 기준으로 책임 조항과 비용 반영 기준을 정리해, 전환 초기의 불안정한 사고 처리 흐름을 빠르게 안정화했습니다.

배상 회수율 14.7배 개선
택배사 회수·사진 증빙 기준 간소화 · 물류 계약서 기반 비용 차감 기준 운영 반영 · 전환 초기 사고 처리 흐름 안정화

Evidence

· 계약서 기준 확인 및 유관 리더 협의 · 월별 피해 현황 집계·정산 참고 자료 공유

Insight

· 관행은 기준이 아니라, 기준이 정리되지 않았을 때 남는 기본값일 수 있습니다.

· 처음 접한 업무 영역이라도 원문과 실제 운영 데이터를 함께 보면, 책임 주체와 비용 반영 기준을 구조화할 수 있습니다.

· 파트너 운영에서 중요한 것은 단순히 보상을 요청하는 것이 아니라, 사고 유형·증빙·정산 기준을 같은 언어로 맞추는 것입니다.

Case 05

AI·자동화 기반 운영 효율화 — 반복 처리 7→1단계 축소와 내부 조회 구조 개선

Problem

대량 AS/재배송·반품 처리 시 동일 항목을 반복 입력해야 했고, 처리 건수가 늘어날수록 입력 피로도와 기준값 오류 가능성이 커졌습니다. 특히 귀책 대상, 처리 사유, 보상 기준처럼 후속 정산·집계에 연결되는 값은 한 번 잘못 입력되면 운영 데이터 전체가 왜곡될 수 있었습니다.

또한 상품 문의나 품질 이슈 처리 시 담당 MD와 상품코드를 확인하려면 매번 엑셀이나 별도 자료를 열어야 했고, 반복 조회와 이관 과정에서 처리 시간이 누적됐습니다.

A 반복 처리 자동화 — 7단계 → 1단계

Claude의 도움을 받아 Chrome Extension을 구현하고, AS/재배송·반품 등록 시 반복 입력되는 주요 항목이 단축키 한 번으로 자동 입력되도록 설계했습니다. 기존에는 주문 확인, 값 입력, 사유 선택, 상태 변경 등 여러 단계를 반복해야 했지만, 반복 입력 구간을 자동화해 처리 동선을 7단계에서 1단계로 줄였습니다.

해당 방식으로 AS/재배송 1,151건 이상을 처리했으며, 이후 반복 업무에도 계속 활용하고 있습니다.

다만 모든 단계를 자동화하지는 않았습니다. 귀책 판단, 예외 케이스 확인, 최종 등록처럼 비용·정산·고객 안내에 영향을 주는 구간은 담당자가 직접 확인하도록 남겨 오류 리스크를 낮췄습니다.

B Slack 기반 내부 조회 구조 구축

상품 문의나 품질 이슈 처리 시 담당 MD와 상품코드를 확인하는 과정도 반복적으로 발생했습니다. 기존에는 엑셀이나 별도 자료를 열어 검색해야 했기 때문에, 문의 이관 전 확인 시간이 계속 누적됐습니다.

Slack API와 Apps Script를 활용해 /담당자 슬래시 커맨드 조회 인터페이스를 구축했습니다. 상품명 일부 또는 상품코드를 입력하면 담당 MD와 상품코드가 즉시 조회되고, 결과는 입력자에게만 노출되도록 구성했습니다.

이를 통해 상품 문의 이관 전 내부 조회 동선을 줄이고, 담당자 확인 기준을 팀원들이 동일하게 사용할 수 있도록 정리했습니다.

C AI를 실무 파트너로 활용

Claude, ChatGPT, NotebookLM은 운영 기준 정리, 자동화 로직 설계 보조, 디버깅, 리스크 표현 검토, VOC 분류와 문서 초안 작성에 활용했습니다.

AI가 만든 결과를 그대로 사용하는 방식이 아니라, 실제 정책·원문·운영 데이터와 대조해 검수하는 방식으로 사용했습니다. 이 활용성을 인정받아 AI 도구를 공식 업무 도구로 승인받아 실무에 적용했습니다.

Result

반복 입력 업무는 7단계에서 1단계로 줄였고, AS/재배송 1,151건 이상을 처리하는 데 활용했습니다. 내부 조회 업무는 Slack 명령어 기반으로 전환해 담당 MD·상품코드 확인 동선을 단축했습니다.

자동화 대상과 사람의 판단이 필요한 구간을 분리해, 처리 속도뿐 아니라 기준값 일관성과 오류 감소까지 함께 개선했습니다.
