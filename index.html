CX는 고객 접점에서 발견한 문제를 제품·운영 구조로 바꾸는 일이라고 생각합니다. 고객 화면만 바꾸는 일이 아니라, 문의 이후 데이터가 어떤 상태로 남고, 누가 확인하며, 어떤 기준으로 고객에게 안내되는지까지 설계하는 일이라고 봅니다.

저는 반복 문의를 단순 처리로 끝내지 않고, 고객이 어느 시점에서 오인하고 내부 처리가 어디서 다시 확인되는지 먼저 봅니다. 배송 제약 VOC는 UX 개선안으로, 환불·교환 문의는 의도 선택값 구조로 연결해왔습니다.

초록마을에서는 B2C 고객 VOC를 제품·운영 개선 과제로 연결했고, 카카오VX에서는 제휴사별로 상이하던 예약·취소·위약 기준을 표준화해 운영 판단 기준으로 정리했습니다. 처음 접하는 도메인도 빠르게 구조화해 고객과 담당자가 이해할 수 있는 운영 언어로 바꾸는 것이 제 강점입니다.

Case 01

배송일 충돌 오인을 줄이기 위한 상품별 안내 구조 개선 — 개발팀 제안·당일 구현

Problem

배송 가능일이 다른 상품이 장바구니에 함께 담기면 기존 시스템 메시지는 일반 품절처럼 보였고, 고객은 실제 품절인지 배송일 충돌인지 구분하기 어려웠습니다. 이로 인해 동일한 문의가 일 6~10건 반복됐습니다.

A 상품별 subtext 개선안 정리 · 개발팀 제안

특정 요일에만 도착 가능한 상품에 상품별 안내 subtext를 노출하도록 개선안을 정리해 데이터 자료와 함께 개발팀에 제안했습니다. 고객이 장바구니에서 배송 제약 상품을 직접 확인할 수 있도록 하는 최소 실행 단위로 접근했습니다.

제안은 즉시 수용되어 당일 구현됐고, 이후 유사한 배송 제약이 있는 상품에도 같은 구조로 확장 적용됐습니다.

Result

고객은 장바구니 단계에서 배송 제약이 있는 상품을 구분할 수 있게 되었고, 반복 문의는 일 6~10건에서 0건 수준으로 감소했습니다.

일 6~10건 → 0건 수준
반복 문의 해소 · 개발팀 제안·당일 구현 · 유사 케이스로 확장 적용

Insight

· 같은 정보를 충분히 안내하지 않을 때, 고객은 시스템 오류로 인식합니다.

· 모든 예외를 분기하기보다, 막히는 시점에 이유를 보여주는 UX가 더 실용적입니다.

· 고객 접점에서 발견한 패턴도 운영 기준이 있으면 제품 표준으로 확장됩니다.

Case 02

고객 의도 기반 환불·교환 입력 구조 재설계 — 재확인 왕복 해소

Problem

환불·교환 등 고객이 원하는 해결방법이 별도 선택값으로 분리되어 있지 않아, 자유 텍스트 안에서 고객 의도를 다시 확인해야 하는 구조였습니다.

A 1:1 문의 접수 단계 '원하는 해결방법' 선택 필드 신설

1:1 문의 화면에 '원하는 해결방법' 선택 필드를 신설해 고객이 문의 작성 시 환불/교환 의사를 먼저 선택하도록 하고, 해당 값이 CRM의 해결방법 필드로 노출되도록 개선안을 정리했습니다.

B 기존 CRM 영역 활용 · 개발 범위 최소화

답변 작성일과 중복되는 '확인일시' 영역을 '해결방법' 필드로 대체하는 방향을 제안해, 별도 개발 범위를 늘리지 않고 구조 변경이 가능한 지점을 확인했습니다.

Result

고객 의도 확인을 위한 추가 답변 왕복을 줄이고, 담당자가 문의 접수 단계에서 해결방법 선택값을 바로 확인할 수 있어 후속 처리 기준이 더 명확해졌습니다.

재확인 왕복 해소 · 처리 기준 명확화
자유 텍스트 → 해결방법 선택값 · CRM 필드 연결 · 개발 범위 최소화

Case 04

외부 물류 파트너 사고·배상 기준 정리 — 배상 회수율 14.7배 개선

Problem

택배사와 3PL 파트너의 사고 확인 흐름, 증빙 기준, 비용 반영 기준이 각각 달라 배상 회수와 후속 처리가 지연되는 문제가 있었습니다. 배상 회수액은 월 50만 원 수준에 머물러 있었습니다.

A 택배 표준약관 근거 제시 · 증빙 기준 간소화

택배 표준약관상 사고 발생 시 조치 의무를 근거로 논점을 정리했습니다. 그 결과 파손 스캔 등 시스템상 사고가 확인된 건은 회수·사진 3종 없이 사고접수가 가능해졌고, 배상 회수액은 월 50만 원 수준에서 약 700만 원 수준으로 개선되었습니다.

B 물류 계약서 기반 물류비 차감 기준 정리

물류 계약서 원문을 확인해 초록마을이 먼저 처리한 환불액, 재배송비, CS 보상 비용을 익월 물류비 정산에서 차감 반영할 수 있는 구조로 정리했습니다.

Result

배상 회수액을 월 50만 원 수준에서 약 700만 원 수준으로 개선했습니다.

Case 05

AI·자동화 기반 운영 효율화 — 반복 처리 7→1단계 축소와 내부 조회 구조 개선

Problem

대량 AS/재배송·반품 처리 시 동일 항목을 반복 입력해야 했고, 처리 건수가 늘어날수록 입력 피로도와 기준값 오류 가능성이 커졌습니다.

A 반복 처리 자동화 — 7단계 → 1단계

Claude의 도움을 받아 Chrome Extension을 구현하고, 처리 동선을 7단계에서 1단계로 줄였습니다. AS/재배송 1,151건 이상을 처리했으며, 리스크 구간은 담당자가 직접 확인하도록 남겨 오류 리스크를 낮췄습니다.

B Slack 기반 내부 조회 구조 구축

Slack API와 Apps Script를 활용해 /담당자 슬래시 커맨드 조회 인터페이스를 구축했습니다.

C AI를 실무 파트너로 활용

AI가 만든 결과를 실제 정책·원문·운영 데이터와 대조해 검수하는 방식으로 사용했습니다. AI 도구를 공식 업무 도구로 승인받아 실무에 적용했습니다.

Result

반복 입력 업무는 7단계에서 1단계로 줄였고, AS/재배송 1,151건 이상을 처리하는 데 활용했습니다.
